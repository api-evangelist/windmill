<h2 class="anchor anchorWithStickyNavbar_LWe7" id="the-problem">The problem<a class="hash-link" href="https://www.windmill.dev/blog/launch-week-ai-sandboxes#the-problem" title="Direct link to The problem">​</a></h2>
<p>AI coding agents need two things that are hard to combine: isolation and persistence. You want them sandboxed so they cannot access the host filesystem or network. But you also want them to remember state across runs, produce artifacts, and pick up where they left off.</p>
<p>Teams end up managing Docker containers, mounting volumes manually, and writing wrapper scripts to handle session state. The orchestration layer has no opinion about where the agent runs or how its files persist.</p>
<h2 class="anchor anchorWithStickyNavbar_LWe7" id="ai-sandboxes-two-annotations">AI sandboxes: two annotations<a class="hash-link" href="https://www.windmill.dev/blog/launch-week-ai-sandboxes#ai-sandboxes-two-annotations" title="Direct link to AI sandboxes: two annotations">​</a></h2>
<p>An AI sandbox is a regular Windmill script with two annotations: one for isolation, one for storage.</p>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="persistent-volumes">Persistent volumes<a class="hash-link" href="https://www.windmill.dev/blog/launch-week-ai-sandboxes#persistent-volumes" title="Direct link to Persistent volumes">​</a></h2>
<p>Agents need to remember state across runs. Volumes handle this by syncing files to your <a href="https://www.windmill.dev/docs/core_concepts/persistent_storage">workspace object storage</a> (S3, Azure Blob, GCS) between executions.</p>
<p>Declaring a volume is a single annotation: <code>// volume: &lt;name&gt; &lt;mount_path&gt;</code>. You can attach up to 10 volumes per script.</p>
<p>Each volume goes through three phases per execution:</p>
<ol>
<li><strong>Before execution</strong>: Windmill acquires an exclusive lease on the volume and downloads files from object storage. A per-worker LRU cache (up to 10 GB) skips the download when files haven't changed (compared by size and MD5).</li>
<li><strong>During execution</strong>: the volume is bind-mounted into the sandbox. The agent reads and writes files normally.</li>
<li><strong>After execution</strong>: changed files are synced back to object storage, metadata is updated, and the lease is released.</li>
</ol>
<p>The exclusive lease (60-second TTL, auto-renewed every 10 seconds) guarantees that only one job writes to a volume at a time. If another job targets the same volume, it waits for the lease to be released.</p>
<p>Volume names support <code>$workspace</code> and <code>$args[...]</code> interpolation, so you can scope storage per workspace, per user, or per any input parameter. This makes it straightforward to give each agent session its own isolated storage.</p>
<p>Volumes also have fine-grained <a href="https://www.windmill.dev/docs/core_concepts/volumes">permissions</a>: owner, read-only, or read-write access, assignable per user or group. A job with no permission on a volume will fail, so you control exactly which agents can access which data.</p>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="process-isolation">Process isolation<a class="hash-link" href="https://www.windmill.dev/blog/launch-week-ai-sandboxes#process-isolation" title="Direct link to Process isolation">​</a></h2>
<p>Without isolation, an agent has access to the host filesystem, environment variables (including credentials), other running jobs, and unrestricted network. Windmill provides two levels of process isolation to prevent this.</p>
<p><strong>NSJAIL sandboxing</strong> is the strongest option. Each execution runs in its own <a href="https://www.windmill.dev/docs/advanced/security_isolation">NSJAIL</a> sandbox with:</p>
<ul>
<li><strong>Filesystem isolation</strong>: the agent only sees its own working directory and mounted volumes. No access to the host filesystem or other jobs.</li>
<li><strong>Network restrictions</strong>: outbound network access can be restricted per sandbox.</li>
<li><strong>Resource limits</strong>: CPU, memory, and disk usage are capped per execution.</li>
</ul>
<p>Enable it per script with the <code>// sandbox</code> annotation, or force it instance-wide for all jobs with <code>DISABLE_NSJAIL=false</code>.</p>
<p><strong>PID namespace isolation</strong> is a lighter alternative for workloads where full sandboxing is unnecessary. It uses Linux <code>unshare</code> to create separate process namespaces, so each job gets its own process tree and cannot see or signal processes from other jobs. Enable it with <code>ENABLE_UNSHARE_PID=true</code>. Lower overhead, but no filesystem or network isolation.</p>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="works-with-any-agent">Works with any agent<a class="hash-link" href="https://www.windmill.dev/blog/launch-week-ai-sandboxes#works-with-any-agent" title="Direct link to Works with any agent">​</a></h2>
<p>Claude Code, Codex, OpenCode, or any custom agent that operates on a local filesystem. Windmill provides the sandbox and the storage; the agent brings its own logic. A built-in Claude Code template handles session persistence and token counting out of the box.</p>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="observability">Observability<a class="hash-link" href="https://www.windmill.dev/blog/launch-week-ai-sandboxes#observability" title="Direct link to Observability">​</a></h2>
<p>Every agent run is a regular Windmill job, which means full <a href="https://www.windmill.dev/platform/observability">observability</a> out of the box: logs, execution history, and token usage for cost monitoring. Set up <a href="https://www.windmill.dev/docs/core_concepts/critical_alerts">alerts</a> on failures or cost thresholds, and audit agent activity across workspaces.</p>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="use-cases">Use cases<a class="hash-link" href="https://www.windmill.dev/blog/launch-week-ai-sandboxes#use-cases" title="Direct link to Use cases">​</a></h2>
<ul>
<li><strong>Persistent agent memory</strong>: conversation history and session state survive across runs.</li>
<li><strong>Artifact generation</strong>: agents produce reports, code, or data files that persist in the volume.</li>
<li><strong>Multi-step workflows</strong>: a flow triggers an agent, waits for results, then passes artifacts to the next step.</li>
<li><strong>Safe execution at scale</strong>: resource limits and isolation let you run untrusted agent code without risk.</li>
</ul>
<h2 class="anchor anchorWithStickyNavbar_LWe7" id="getting-started">Getting started<a class="hash-link" href="https://www.windmill.dev/blog/launch-week-ai-sandboxes#getting-started" title="Direct link to Getting started">​</a></h2>
<ol>
<li>Configure <a href="https://www.windmill.dev/docs/core_concepts/persistent_storage">workspace object storage</a> (S3, Azure Blob, GCS, or filesystem).</li>
<li>Add <code>// sandbox</code> and <code>// volume: &lt;name&gt; &lt;path&gt;</code> annotations to any script.</li>
<li>Run it. Files in the volume path persist across executions.</li>
</ol>
