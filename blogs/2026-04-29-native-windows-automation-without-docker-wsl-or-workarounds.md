<p>You manage Windows servers, Active Directory, MSSQL, and PowerShell runbooks. You need to automate server provisioning, user management, database maintenance, and compliance checks. You also need version control, approval workflows, and audit trails around all of it.</p>
<p>The usual answer is "install Docker" or "set up WSL2." But your PowerShell scripts need access to AD, your MSSQL queries use Kerberos tickets from the domain, and your modules come from private Azure Artifacts feeds. Containerizing all of that means fighting your own infrastructure.</p>
<p>Windmill runs as a native Windows service. <code>windmill-ee.exe</code> registers via <code>sc.exe</code> and executes PowerShell, C#, and SQL directly on your domain-joined servers. This post walks through what that enables: typed PowerShell with <code>[CmdletBinding()]</code>, MSSQL with Kerberos auth, Teams approvals, and AI agents that use all of the above as tools.</p>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="native-windows-workers">Native Windows workers<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#native-windows-workers" title="Direct link to Native Windows workers">​</a></h2>
<p>Windmill's Windows worker is a single executable: <code>windmill-ee.exe</code>. It runs anywhere Windows runs: servers, workstations, laptops, edge devices. Three modes:</p>
<ul>
<li><strong>Worker</strong>: connects directly to the Windmill database, executes jobs</li>
<li><strong>Server</strong>: runs the Windmill API and web UI</li>
<li><strong>Agent</strong>: connects to a remote Windmill server over HTTPS, with no direct database access. Ideal for workers behind firewalls. See <a href="https://www.windmill.dev/docs/core_concepts/agent_workers">agent workers</a> for how to generate tokens</li>
</ul>

<h3 class="anchor anchorWithStickyNavbar_LWe7" id="mixed-windowslinux-fleet">Mixed Windows/Linux fleet<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#mixed-windowslinux-fleet" title="Direct link to Mixed Windows/Linux fleet">​</a></h3>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="powershell-as-a-first-class-language">PowerShell as a first-class language<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#powershell-as-a-first-class-language" title="Direct link to PowerShell as a first-class language">​</a></h2>

PowerShell scripts get the same capabilities as Python or TypeScript: schedules, webhooks, approval flows, error handling, and composition in <a href="https://www.windmill.dev/docs/flows/flow_editor">flows</a>.</p>

<h3 class="anchor anchorWithStickyNavbar_LWe7" id="cmdletbinding-and-common-parameters">CmdletBinding and common parameters<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#cmdletbinding-and-common-parameters" title="Direct link to CmdletBinding and common parameters">​</a></h3>
<p>When a script declares <code>[CmdletBinding()]</code>, Windmill shows toggles for <code>-Verbose</code>, <code>-Debug</code>, and an <code>-ErrorAction</code> dropdown in the run form. These are the same common parameters you'd use in a PowerShell console. <code>Write-Verbose</code> and <code>Write-Debug</code> output appears in job logs with <code>VERBOSE:</code> / <code>DEBUG:</code> prefixes. Scripts without <code>[CmdletBinding()]</code> are unaffected.</p>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="mssql-with-kerberos">MSSQL with Kerberos<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#mssql-with-kerberos" title="Direct link to MSSQL with Kerberos">​</a></h2>
<p>Windmill's <a href="https://www.windmill.dev/docs/integrations/mssql">MSSQL integration</a> supports three auth methods: SQL auth (username/password), Azure AD/Entra (OAuth), and Windows Integrated Authentication (Kerberos).</p>
<p>With Kerberos, the worker's service account credentials are used directly, with no database passwords stored in Windmill. The worker inherits its domain identity, and MSSQL trusts it through AD.</p>
<p>Requirements:</p>
<ul>
<li>Valid Kerberos ticket on the worker (via <code>kinit</code> or a keytab)</li>
<li>Correct <code>krb5.conf</code> realm configuration</li>
<li>Service account permissions on the target database</li>
</ul>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="ai-agents-on-windows-infrastructure">AI agents on Windows infrastructure<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#ai-agents-on-windows-infrastructure" title="Direct link to AI agents on Windows infrastructure">​</a></h2>
<p>Running PowerShell against a domain-joined server or querying MSSQL with Kerberos auth isn't something most AI agent platforms can do.</p>
<p>Windmill's <a href="https://www.windmill.dev/docs/core_concepts/ai_agents">AI agent</a> flow steps use scripts as tools. Any Windmill script (PowerShell, C#, SQL, TypeScript) becomes a callable tool where typed parameters map to the tool's JSON schema. The agent sees what each tool does and what inputs it needs, then calls them to accomplish a goal.</p>

<li>A Teams notification script for escalation</li>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="replacing-task-scheduler-and-power-automate">Replacing Task Scheduler and Power Automate<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#replacing-task-scheduler-and-power-automate" title="Direct link to Replacing Task Scheduler and Power Automate">​</a></h2>
<p>The typical migration path: PowerShell scripts scattered across servers on Task Scheduler, Power Automate for approvals, alerts via email, no audit trail.</p>
<p>In Windmill, each script becomes version-controlled, gets an auto-generated UI, and can be composed into flows with approval steps (via <a href="https://www.windmill.dev/docs/integrations/teams">Teams</a>), error handling, and retries. Windows and Linux workers run in the same instance. Every execution is logged with inputs, outputs, duration, and who triggered it.</p>
<p>The migration is incremental: start with one runbook, schedule it in Windmill, and let the Task Scheduler job run in parallel until you're confident.</p>
<h2 class="anchor anchorWithStickyNavbar_LWe7" id="how-windmill-compares-on-windows">How Windmill compares on Windows<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#how-windmill-compares-on-windows" title="Direct link to How Windmill compares on Windows">​</a></h2>
<p><a href="https://airflow.apache.org/" rel="noopener noreferrer" target="_blank">Airflow</a> has had an <a href="https://github.com/apache/airflow/issues/10388" rel="noopener noreferrer" target="_blank">open issue for Windows support</a> since 2020, and it's not on the roadmap. <a href="https://www.prefect.io/" rel="noopener noreferrer" target="_blank">Prefect</a> workers can run on Windows via pip, but the docs focus on Docker/Kubernetes and there's no service management. <a href="https://n8n.io/" rel="noopener noreferrer" target="_blank">n8n</a> runs well via Node.js but is low-code and connector-focused. <a href="https://kestra.io/" rel="noopener noreferrer" target="_blank">Kestra</a> is the closest: it runs as a JAR, has a PowerShell plugin, and supports a PROCESS runner, but lacks typed parameter inference, private Azure Artifacts, MSSQL Kerberos, and C# support.</p>
<p>Power Automate has strong Teams integration, but no self-hosted option, no version control, no code-first workflow definition, and premium licensing at $15/user/month. That means no GitOps, no PR reviews on flow changes, and no self-hosted deployment.</p>

<h2 class="anchor anchorWithStickyNavbar_LWe7" id="getting-started">Getting started<a class="hash-link" href="https://www.windmill.dev/blog/windows-workflow-engine#getting-started" title="Direct link to Getting started">​</a></h2>
<p>The Windmill server runs in Docker (Linux). Windows workers connect as native agents, with no Docker on the Windows side.</p>
<ol>
<li>Deploy the <a href="https://www.windmill.dev/docs/advanced/self_host">Windmill server</a> (Docker Compose or Kubernetes)</li>
<li>Download <code>windmill-ee.exe</code> on your Windows server</li>
<li>Register it as a Windows service in agent mode</li>
<li>Connect with an agent token</li>
</ol>
<p>Windows workers require an <a href="https://www.windmill.dev/pricing">Enterprise license</a>.</p>
