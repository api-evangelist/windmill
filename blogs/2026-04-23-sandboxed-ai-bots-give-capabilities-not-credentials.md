<p>The <a href="https://vercel.com/kb/bulletin/vercel-april-2026-security-incident" rel="noopener noreferrer" target="_blank">Vercel security incident</a> earlier this month leaked customer environment variables, including API keys for payment processors and databases. A month before that, <a href="https://docs.litellm.ai/blog/security-update-march-2026" rel="noopener noreferrer" target="_blank">compromised LiteLLM packages on PyPI</a> stole AWS, GCP, and SSH credentials from developer machines. Secrets stored as env vars or in plaintext keep ending up in the wrong hands.</p>
<p>If you're building AI agents that call sensitive APIs, the question is: does the agent actually need to see the API key, or just the ability to call the API?</p>

<p>Someone asks "did Acme pay?" and the agent calls <code>get_customer</code>, then <code>search_invoices</code>, then responds in plain English. It decides which tools to call based on the question, but it never sees the Stripe secret key.</p>
<h2 class="anchor anchorWithStickyNavbar_LWe7" id="how-credentials-stay-isolated">How credentials stay isolated<a class="hash-link" href="https://www.windmill.dev/blog/discord-bot-stripe-sandboxed-ai#how-credentials-stay-isolated" title="Direct link to How credentials stay isolated">​</a></h2>
<p>The tool implementation fetches secrets from <a href="https://www.windmill.dev/docs/core_concepts/resources_and_types">Windmill resources</a> at runtime. The agent only sees the tool signature ("give me an email, I'll return invoice data"). The credential itself lives in Windmill's encrypted resource store and is injected inside the tool function, not passed to the agent.</p>
<p>In the typical setup, <code>STRIPE_SECRET_KEY</code> sits in an env var and the whole process has access to it. In Windmill, <code>process.env</code> in the agent's sandbox only contains runtime variables like workspace ID and job ID. No secrets.</p>
<h2 class="anchor anchorWithStickyNavbar_LWe7" id="the-sandbox">The sandbox<a class="hash-link" href="https://www.windmill.dev/blog/discord-bot-stripe-sandboxed-ai#the-sandbox" title="Direct link to The sandbox">​</a></h2>
<p>Windmill's <a href="https://www.windmill.dev/blog/launch-week-ai-sandboxes">AI sandboxes</a> (introduced during <a href="https://www.windmill.dev/launch-week-march-2026">launch week</a>) use <a href="https://www.windmill.dev/docs/advanced/security_isolation#nsjail-sandboxing">nsjail</a> for process-level isolation:</p>
<ul>
<li><strong>Credential isolation</strong>: API keys live in the encrypted <a href="https://www.windmill.dev/docs/core_concepts/resources_and_types">resource store</a>, not in the agent's environment. Windmill supports <a href="https://www.windmill.dev/docs/core_concepts/variables_and_secrets">multiple secret storage backends</a>: its own encrypted store, HashiCorp Vault, Azure Key Vault, and AWS Secrets Manager. Either way, the agent never sees them.</li>
<li><strong>Filesystem isolation</strong>: scripts run in an isolated mount namespace with access only to their job directory and explicitly mounted <a href="https://www.windmill.dev/docs/core_concepts/volumes">volumes</a>.</li>
<li><strong>Resource limits</strong>: CPU, memory, and process limits per job.</li>
<li><strong>Network isolation</strong> (optional): outbound network access is on by default (tools need it to call APIs). For stricter setups, you can enable network namespace isolation via <a href="https://www.windmill.dev/docs/advanced/security_isolation#pid-namespace-isolation">unshare flags</a> (<code>--net</code>) to block all outbound connections.</li>
</ul>
<p>Even without network isolation, the agent can't authenticate to Stripe or any other service because it doesn't have the key. It can only use what you explicitly wire up as tools.</p>
